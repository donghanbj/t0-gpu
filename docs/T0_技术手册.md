# T0-GPU 技术手册

> 纯 Rust 裸金属 GPU 内核编译器 & KFD 运行时 — AMD RDNA3 (GFX1100)
> 版本: 2026-03-29 | 代码总量: ~50,000 LOC Rust

---

## 1. 项目概述

T0-GPU 是一个**完全自包含**的 GPU 编程框架：

- **输入**: Rust DSL 描述的数学运算
- **输出**: AMD HSA ELF 二进制 + KFD 裸金属调度
- **零外部依赖**: 不依赖 HIP/ROCm/Triton/LLVM-IR，仅需 `/dev/kfd` 驱动接口

### 1.1 核心价值

| 对比维度 | Triton on AMD | T0-GPU |
|---------|:------------:|:------:|
| 性能 (4096³ GEMM) | ~80 TFLOPS | **~79 TFLOPS** |
| 依赖栈 | Python + LLVM + ROCm | **仅 Rust + /dev/kfd** |
| 编译延迟 | 秒级 (LLVM multi-pass) | **毫秒级** (直接 ISA) |
| Dispatch 延迟 | ~10-50μs (HIP) | **~2μs** (AQL doorbell) |
| 二进制大小 | Triton+LLVM+ROCm = 数 GB | **单一 Rust binary** |

### 1.2 代码量统计

| 模块 | 文件 | 行数 | 说明 |
|------|------|-----:|------|
| T0 编译器 | `src/t0/*.rs` (34 文件) | ~38,000 | 编译流水线全部代码 |
| ISA 编码器 | `rdna3_asm.rs` | 3,114 | GFX1100 机器码编码 |
| ELF 生成器 | `rdna3_code_object.rs` | 1,444 | AMD HSA Code Object |
| KFD 运行时 | `kfd/mod.rs` | 3,021 | 裸金属 GPU 控制 |
| **总计** | **38 文件** | **~50,000** | |

### 2.2 两条编译路径

T0 有两条独立的编译路径，针对不同类型的内核优化：

**Path A: BlockDSL → SSA Pipeline** (通用内核)
- 适用于：逐元素运算、Softmax、RoPE、Cross-Entropy Loss 等
- 流程：`BlockKernel` DSL → `SsaFunc` SSA IR → 优化 Pass → SSA RegAlloc → `T0Kernel`
- 特点：通用性强，支持控制流、条件分支、循环

**Path B: TileIR → Tile SSA Pipeline** (GEMM 专用)
- 适用于：矩阵乘法 (WMMA)、Causal GEMM
- 流程：`GemmConfig` → `TileFunc` → `TileSsaFunc` → Lower → `T0Kernel`
- 特点：GEMM 专用优化（cooperative load、LDS double-buffer、K-loop 流水线）

## 3. 模块详解

### 3.1 T0 IR (`ir.rs`, 1,140 行)

T0 的中间表示层，定义了所有虚拟寄存器和操作类型。

**虚拟寄存器体系**:
```rust
VReg(u32)     // 虚拟 VGPR（向量寄存器）
SReg(u32)     // 虚拟 SGPR（标量寄存器）
SRegPair(u32) // 64-bit 标量指针对 (s[n:n+1])
```

**对齐约束** (`Alignment`):
- `None`: 无约束
- `Align2`: 2-对齐 (SGPR pair, dwordx2)
- `Align4`: 4-对齐 (dwordx4, SMEM s_load_b128)
- `Align8`: 8-对齐 (**WMMA fragment — 8 连续 VGPRs**)

**操作类型** (`Op` enum, ~80 种):
- 全局内存: `GlobalLoad`, `GlobalStore`, `BufferLoad`, `BufferStore`
- LDS: `LdsLoad`, `LdsStore`, `DsLoadB128`, `DsStoreB128` 等
- VALU: `VAddF32`, `VMulF32`, `VFmaF32`, `VMaxF32`, `VSubF32` 等
- WMMA: `Wmma { dst, a, b, c, format }` (bf16→f32, f16→f32, bf16→bf16)
- 标量 ALU: `SAddU32`, `SMulI32`, `SLshlB32`, `SCmpGeU32` 等
- 控制流: `Label`, `Branch`, `BranchScc1`, `BranchVccz`
- EXEC mask: `SaveExec`, `RestoreExec`, `XorExec`
- 数据转换: `CvtPkBf16F32` (f32→bf16x2 打包)
- 跨 lane: `DsSwizzle`, `VPermlanex16B32`, `WaveReduceAddF32`
- 同步: `Barrier`, `WaitVmcnt`, `WaitLgkmcnt`, `WaitVscnt`
- 原子: `GlobalAtomicAddF32`, `GlobalAtomicAddU32Rtn`

每个 `Op` 实现三个关键方法:
- `vreg_refs()`: 所有引用的 VReg (def + use)
- `vreg_defs()`: 定义（写入）的 VReg
- `vreg_uses()`: 使用（读取）的 VReg

### 3.2 BlockDSL (`block_dsl.rs`, 2,076 行)

声明式内核构建 DSL，用于非 GEMM 内核。

```rust
let mut kb = BlockKernel::new("vadd", 256); // name, workgroup_size
let x = kb.arg_ptr("x");                    // 声明 kernarg 指针
let n = kb.arg_u32("n_elems");              // 声明 kernarg 标量
let gid = kb.global_id();                   // v_mad_u32 计算全局 ID

kb.if_lt(gid, n, |kb| {                     // EXEC mask 边界检查
    let val = kb.load_f32(x, gid);          // global_load_b32
    let result = kb.mul(val, kb.const_f32(2.0));
    kb.store_f32(x, gid, result);           // global_store_b32
});
let compiled = kb.compile(Target::GFX1100)?;
```

**支持的构造**:
- 指针算术 (`ptr_offset_bytes`)、标量 kernarg (`arg_u32`, `arg_f32`)
- 循环 (`for_range`)、条件 (`if_lt`, `if_then_else`)
- LDS 操作 (`lds_alloc`, `lds_load`, `lds_store`, `lds_barrier`)
- WMMA (`wmma_bf16_f32`)
- Wave-level reduce (`wave_reduce_add_f32`, `wave_reduce_max_f32`)
- BF16 打包 (`cvt_pk_bf16_f32`)
- 原子操作 (`atomic_add_f32_global`)

### 3.3 BlockDSL → SSA (`block_dsl_to_ssa.rs`, 1,787 行)

将 BlockDSL 的高级结构化指令降级为 SSA IR：
- 展开 `for_range` 为 SALU 循环 (s_add_u32 + s_cbranch_scc1)
- 展开 `if_lt` 为 EXEC mask save/restore
- 展开 `wave_reduce_add_f32` 为 5 级 ds_swizzle XOR 蝶形规约
- 展开 `cvt_pk_bf16_f32` 为 v_lshrrev_b32 + v_and_or_b32 序列

### 3.4 SSA IR (`ssa_ir.rs`, 3,383 行)

SSA (Static Single Assignment) 形式的中间表示：

```rust
struct SsaFunc {
    blocks: Vec<SsaBlock>,    // 基本块列表
    entry: BlockId,            // 入口块
}
struct SsaBlock {
    id: BlockId,
    ops: Vec<SsaOp>,          // SSA 操作序列
    terminator: Terminator,    // 分支/返回
}
```

**SSA 操作类型** (`SsaOp`): 与 `ir::Op` 类似但使用 `MVal` (SSA 值标号) 代替 `VReg`。每个值只被定义一次，支持 SSA 图分析。

**关键扩展**:
- `Phi` 节点 (合并不同路径的值)
- 群组约束 (`GroupDef`: WMMA 8-VGPRs, dwordx4 4-VGPRs)
- 控制流图 (CFG: 前驱/后继边)

### 3.5 SSA 优化 Pass (`opt_passes.rs`, 1,362 行)

6 个优化 Pass，按依赖顺序执行：

| Pass | 说明 | 效果 |
|------|------|------|
| **DCE** | Dead Code Elimination | 删除未使用的计算 |
| **CSE** | Common Subexpression Elimination | 合并重复的标量计算（barrier-aware） |
| **LICM** | Loop-Invariant Code Motion | 将不变量移出循环体 |
| **Copy Propagation** | 消除 v_mov 拷贝链 | 减少寄存器压力 |
| **Algebraic Simplification** | x+0=x, x*1=x, x*0=0 | 减少指令数 |
| **Waitcnt Refinement** | 精化 wait_vmcnt/lgkmcnt 计数 | 减少不必要的 stall |

**CSE 的特殊处理**: GEMM 内核中 CSE 跨 barrier 合并会导致数据竞争，因此通过 `skip_cse` 标志禁用。

### 3.6 SSA 寄存器分配 (`ssa_regalloc.rs`, 1,040 行)

基于 SSA 活跃区间的线性扫描寄存器分配器：

**分配策略**:
1. 计算每个 `MVal` 的活跃区间 `[def_point, last_use]`
2. 按 def_point 排序
3. 高水位标记 (HWM) + FreePool 混合分配：
   - 优先从 FreePool 回收已死亡的寄存器
   - FreePool 空时从 HWM 向上分配新寄存器
4. 群组分配：WMMA 8-aligned, dwordx4 4-aligned

**关键机制**:
- **Gap Reclaim**: 对齐跳过的间隙 VGPRs 回收到 FreePool（节省 ~15 VGPRs）
- **FreePool 传递合并**: 相邻空闲块完整合并，消除碎片化
- **溢出保护**: `max_vgprs = 254`（CWSR 硬件安全上限）
- **Spill 检测**: 超出上限时使用 scratch memory（当前报告但不阻塞编译）

### 3.7 Dominator Tree (`domtree.rs`, 599 行)

Cooper-Harvey-Kennedy 算法实现的支配树，用于：
- SSA Phi 节点放置 (dominance frontier)
- 循环检测 (back-edge → loop-carried liveness)
- LICM 的不变量识别

### 3.8 T0Kernel 编译器 (`compile.rs`, 1,351 行)

编译主入口，负责：

```rust
pub struct T0Kernel {
    name: String, ops: Vec<Op>, args: Vec<KernArg>,
    vreg_allocs: Vec<VRegAlloc>, sreg_allocs: Vec<SRegAlloc>,
    next_vreg: u32, next_sreg: u32,
    kernarg_size: u32, lds_size: u32, wg_size: u32,
    coalesced_groups: Vec<CoalescedGroup>,
    // ...
}
```

**编译流程**:
1. 虚拟寄存器分配 (RegAlloc 或 SSA RegAlloc)
2. AsmEmitter → Rdna3Assembler (生成机器码字节)
3. Rdna3CodeObject → AMD HSA ELF
4. 可选: `T0_DUMP_ASM=1` 导出人类可读 ISA

**关键配置**:
- `use_ssa_regalloc`: 使用 SSA 分配器 (默认 true for BlockDSL path)
- `skip_cse`: 跳过 CSE (GEMM 内核必须)
- `wgp_mode`: 启用 WGP 模式 (2 CU 共享 128KB LDS)

### 3.9 汇编发射器 (`asm_emitter.rs`, 1,022 行)

将 T0 IR `Op` 翻译为 GFX1100 ISA 机器码：

**翻译示例**:
```
Op::VAddF32 { dst, src0, src1 }
  → v_add_f32 v{phys_dst}, v{phys_src0}, v{phys_src1}
  → rdna3_asm::gfx11::v_add_f32(dst, src0, src1)  // 返回 u32

Op::Wmma { dst, a, b, c, BF16_F32 }
  → v_wmma_f32_16x16x16_bf16 v[dst:dst+7], v[a:a+7], v[b:b+7], v[c:c+7]
  → rdna3_asm::gfx11::v_wmma_f32_16x16x16_bf16(dst, a, b, c)  // 返回 [u32; 2]
```

**指令宽度规则**:
| 返回类型 | emit 方法 | 指令类型 |
|---------|----------|---------|
| `u32` | `asm.emit()` | VOP1, VOP2, VOPC, SOPP, SOP1 |
| `[u32; 2]` | `asm.emit2()` | SMEM, FLAT/Global, VOP3, DS, WMMA |

### 3.10 ISA 编码器 (`rdna3_asm.rs`, 3,114 行)

`gfx11` 模块直接编码 GFX1100 机器码，所有编码经 `llvm-mc` 验证。

**覆盖的指令格式**:
- **SOPP**: s_endpgm, s_barrier, s_waitcnt, s_branch, s_cbranch_scc0/1, s_nop, s_clause, s_setprio
- **SOP1**: s_mov_b32 (含 literal, exec_lo 变体)
- **SOP2**: s_add_u32, s_sub_u32, s_addc_u32, s_mul_i32, s_and_b32, s_xor_b32, s_lshl_b32, s_lshr_b32, s_cselect_b32
- **SOPC**: s_cmp_eq_u32, s_cmp_lt_u32, s_cmp_ge_u32, s_cmp_lg_u32, s_cmp_gt_i32
- **SMEM**: s_load_dword, s_load_dwordx2, s_load_dwordx4
- **VOP1**: v_mov_b32, v_rsq_f32, v_exp_f32, v_sin_f32, v_cos_f32, v_rcp_f32, v_sqrt_f32, v_log_f32, v_cvt_f32_u32, v_cvt_u32_f32, v_readfirstlane
- **VOP2**: v_add_f32, v_mul_f32, v_max_f32, v_min_f32, v_add_u32, v_and_b32, v_or_b32, v_xor_b32, v_sub_f32, v_sub_u32, v_lshlrev_b32, v_lshrrev_b32, v_cndmask_b32, v_mul_lo_u32
- **VOP3**: v_fma_f32, v_and_or_b32, v_permlanex16_b32, v_add_co_u32, v_add_co_ci_u32
- **VOPC**: v_cmp_lt_u32, v_cmp_ge_u32, v_cmp_gt_f32, v_cmp_eq_u32, v_cmp_ge_i32
- **VOP3P/WMMA**: v_wmma_f32_16x16x16_bf16, v_wmma_f32_16x16x16_f16, v_wmma_bf16_16x16x16_bf16
- **DS (LDS)**: ds_load_b32/b64/b128, ds_store_b32/b64/b128, ds_load_u16, ds_load_u16_d16/d16_hi, ds_load_2addr_b32, ds_store_b16, ds_swizzle_b32
- **FLAT/Global**: global_load_dword/x2/x4, global_store_dword/x2/x4, global_load_ushort, global_store_short, global_atomic_add_f32
- **MUBUF**: buffer_load_b32/b64/b128, buffer_store_b32/b64/b128

**关键编码规则**:
```
VOP1/VOP2 SRC0: VGPR = 256 + vgpr_num
SGPR: 直接使用 0..127
内联常量: 128 + value (0..64), 0xC0 + |value| (负数)
浮点常量: 0xF0=0.5, 0xF2=1.0, 0xF4=2.0, 0xF6=4.0
Literal: src=0xFF + 第二个 dword
```

### 3.11 Code Object 生成器 (`rdna3_code_object.rs`, 1,444 行)

手工构建 AMD HSA ELF 二进制，完全不依赖 LLVM linker。

**ELF 结构**:
```
ELF64 Header (EM_AMDGPU, ELFOSABI_AMDGPU_HSA, ELFABIVERSION_V5)
  ├── Program Headers (PT_LOAD + PT_NOTE)
  ├── .text (KernelDescriptor 64B + ISA 机器码)
  ├── .rodata (常量数据)
  ├── .note (AMDGPU 元数据: amdhsa.version, amdhsa.kernels)
  ├── .symtab (kernel_name + kernel_name.kd)
  ├── .strtab, .shstrtab
  └── .dynsym, .dynstr, .hash, .dynamic (v2 版本)
```

**KernelDescriptor (64 字节)** — 关键字段:
- `COMPUTE_PGM_RSRC1`: VGPR 粒度 (Wave32: 8), FLOAT_MODE (0xCF), IEEE_MODE
- `COMPUTE_PGM_RSRC2`: USER_SGPR=2, TGID_X/Y/Z_EN, LDS_SIZE=0 (GFX11 必须为 0)
- `COMPUTE_PGM_RSRC3`: INST_PREF_SIZE (指令预取大小, 非零关键!)
- `KERNEL_CODE_PROPERTIES`: KERNARG_PTR + WORKITEM_ID_X + WAVEFRONT_SIZE32

提供两个版本:
- `to_bytes()`: 最小 ELF (足够 KFD 加载)
- `to_bytes_v2()`: 含 dynamic sections (兼容 hipModuleLoadData)

## 4. GEMM 编译流水线

### 4.1 参数化 GEMM 生成器 (`gemm_gen.rs`, 1,361 行)

通过 `GemmConfig` 覆盖所有 tile/K/split 组合:

```rust
pub struct GemmConfig {
    pub tile_m: u32,       // 矩阵 M 方向 tile 大小 (64, 128)
    pub tile_n: u32,       // 矩阵 N 方向 tile 大小 (64, 128)
    pub tile_k: u32,       // K 方向 tile 深度 (16, 32, 64)
    pub split_k: u32,      // K 维并行度 (1, 2, 4...)
    pub wg_size: u32,      // 工作组大小 (通常 128 或 256)
    pub wgp_mode: bool,    // WGP 模式 (2 CU 共享)
    pub transpose: GemmTranspose, // 转置模式
}
```

**`auto_select(M, K, N)`**: 根据矩阵形状自动选择最优配置:
- 优先 128×128 (大方阵)
- 瘦矩阵用 128×64 或 64×128
- 自动计算 split_k 以最大化 CU 占用
- clamp_sk() 确保 K 可被 tile_k × split_k 整除

### 4.2 TileIR (`tile_ir.rs`, 3,996 行)

GEMM 专用的 Tile-level IR, 直接描述 WMMA 数据流:

**K-loop 结构** (LDS Double Buffering):
```
Prologue: GMEM→buf0 (cooperative load first tile)
K-loop Phase A: 
  - GMEM→tmp (load next tile async)
  - WMMA compute from buf0
  - tmp→buf1 (store to LDS)
  - barrier
K-loop Phase B:
  - GMEM→tmp (load next tile async)
  - WMMA compute from buf1
  - tmp→buf0 (store to LDS)
  - barrier
Epilogue: WMMA compute last buffer
Store: C→GMEM
```

**Cooperative Loading** (emit_coop_load_buffer):
- 工作组内所有线程协作加载一个 tile
- 每线程加载 16 字节 (1 个 global_load_b128)
- `chunks_per_row = tile_k * 2 / 16` **必须是 2 的幂** (铁律!)
- tid→(row, col) 分解使用 bitwise AND/shift (非 modulo)

**WMMA 计算** (emit_lds_read_and_wmma):
- 流式模式 (n_col_tiles > 4): 逐列 LDS 加载 + WMMA
- 批量模式 (n_col_tiles ≤ 4): 一次性加载所有 frag_b
- 8 列 streaming + 2 k_sub → 16 次 WMMA per K-tile

### 4.3 Tile SSA (`tile_ssa.rs`, 2,077 行)

TileIR 的 SSA 表示, 支持:
- Tile-level 优化 (冗余 LDS load 消除)
- VGPR 压力估算 (编译时 max_vgprs 检查)
- 自动 acc_swap 检测 (累加器在两组 8-VGPRs 间交替)

### 4.4 Tile SSA Lower (`tile_ssa_lower.rs`, 2,317 行)

将 Tile SSA 降级为 T0Kernel 的标准 `Vec<Op>`:
- 生成 cooperative load 地址计算 (SALU: tid decomposition)
- 生成 LDS 地址计算 (VALU: base + tid_row * stride + tid_col * 16)
- 生成 WMMA 指令序列 + LDS fence
- 生成 C-tile store (每线程 store 8×f32 到 GMEM)

### 4.5 自动调优 (`autotune_oracle.rs`, 246 行 + `cost_model.rs`, 932 行)

**Cost Model**: 估算每种 tile 配置的执行时间:
- VALU/WMMA 计算量
- 全局内存带宽需求
- LDS 带宽瓶颈
- 占用率限制 (VGPR/LDS/Wave)

**Autotune Oracle**: 存储 rocBLAS/Triton 的已知性能数据,
作为 T0 性能评估的参照基线。

## 5. KFD 裸金属运行时

### 5.1 设备管理 (`KfdDevice`)

```rust
let device = KfdDevice::open()?;  // 全局单例 (OnceLock)
```

**初始化流程**:
1. `open("/dev/kfd")` (带 5 次重试, 处理 MODE1 Reset 恢复)
2. `IOC_GET_VERSION` → 获取 KFD 版本
3. 自动探测 `gpu_id` (sysfs topology)
4. `IOC_ACQUIRE_VM` (绑定 DRM fd)
5. `IOC_RUNTIME_ENABLE` (KFD ≥ 1.14 必需)
6. GPU 健康探测 (写入/回读验证)

**KFD 安全机制**:
- `ignore_sigpipe()`: 防止 `cargo test | head` 导致队列泄漏 → 后续 hang
- `open_kfd_with_retry()`: GPU Reset 后自动等待恢复
- `gpu_health_probe()`: 检测 VRAM 不稳定状态

### 5.2 内存管理 (`GpuBuffer`)

```rust
let buf = device.alloc_vram(size)?;   // VRAM (GPU 本地)
let buf = device.alloc_gtt(size)?;    // GTT (CPU-GPU 共享, 缓存一致)
let buf = device.alloc_code(size)?;   // 可执行 VRAM (加载内核代码)
let buf = device.alloc_uncached(size)?; // 非缓存 GTT (ring buffer, 信号)
```

**GpuBuffer 特性**:
- CPU/GPU 统一地址 (`host_ptr` = `gpu_addr` via MAP_FIXED)
- RAII: Drop 时自动 UNMAP + FREE
- `write()`, `read_val::<T>()`, `write_val::<T>()`

### 5.3 AQL 队列 (`AqlQueue`)

```rust
let queue = device.create_queue()?;  // 4MB ring, 64K packets
queue.submit(&kernel, [grid_x, grid_y, grid_z], kernarg_addr);
queue.wait_idle()?;  // 轮询 read_ptr == write_ptr
```

**AQL Dispatch Packet** (64 字节硬件格式):
```
header (type + barrier + fence) | setup (dimensions)
workgroup_size [x, y, z]
grid_size [x, y, z]
private_segment_size | group_segment_size (LDS)
kernel_object (VA of KernelDescriptor)
kernarg_address (VA of kernargs)
completion_signal
```

**Doorbell 机制**:
1. 写 AQL packet 到 ring buffer
2. 原子写 write_ptr_va (增加 1 packet)
3. 写 doorbell → CP/MEC 开始解析 packet → GPU 执行

### 5.4 Dispatch Pool (`DispatchPool`)

预分配 kernarg 内存池, 零分配 dispatch:

```rust
let pool = DispatchPool::new(&device, 64)?;  // 64 个 slot
let ka_addr = pool.write_kernargs(slot, &ka_bytes);
queue.submit(&kernel, grid, ka_addr);
```

### 5.5 GpuKernel 加载

```rust
let kernel = GpuKernel::load(&device, &elf_bytes, &KernelLoadConfig {
    workgroup_size: [128, 1, 1],
    lds_size: 32768,
})?;
```

加载流程:
1. 解析 AMD HSA ELF: 找到 `.text` 中的 KernelDescriptor
2. 分配可执行 VRAM, 写入 KD + ISA 代码
3. HDP flush (确保 I-cache 一致性)
4. 返回 `GpuKernel { descriptor_va, ... }`

## 8. 硬件铁律 (GFX1100 / RDNA3)

### 8.1 寄存器约束

| 约束 | 值 | 违反后果 |
|------|------|---------|
| VGPR 上限 | **254** (安全) / 256 (硬件) | CWSR hang |
| SGPR 上限 | 106 (可用) | 编译错误 |
| WMMA 对齐 | 8-VGPR aligned | 静默错误 |
| SMEM SBASE | 偶数对齐 (s[0:1], s[2:3]...) | 硬件异常 |

### 8.2 内存约束

| 约束 | 值 | 说明 |
|------|------|------|
| LDS/CU (CU mode) | **64 KB** | 不可满载使用 (hang风险) |
| LDS/WGP (WGP mode) | 128 KB | 目前 WGP dispatch 有 page fault |
| Global offset | 13-bit signed | -4096 ~ +4095 |
| DS offset | 16-bit unsigned | 0 ~ 65535 |
| Buffer offset | 12-bit unsigned | 0 ~ 4095 |

### 8.3 指令编码铁律

| 铁律 | 说明 |
|------|------|
| **LLVM 验证优先** | 任何新指令编码必须 `llvm-mc -mcpu=gfx1100` 验证 |
| **GFX10≠GFX11** | global_load_ushort: 0x42(GFX10) vs 0x4A(GFX11) |
| **Coop load cpr=2^n** | `chunks_per_row` 必须是 2 的幂 (assert 保护) |
| **v_exp/v_log** | 计算 2^x / log₂(x), **不是** e^x / ln(x) |
| **ds_swizzle XOR** | 16→0x401F, 8→0x201F, 4→0x101F, 2→0x081F, 1→0x041F |

### 8.4 同步铁律

| 铁律 | 说明 |
|------|------|
| **vmcnt → loads only** | global_store 需要 **vscnt** 等待! |
| **barrier 前 lgkmcnt(0)** | LDS write 必须完成才能 barrier |
| **EXEC restore** | SaveExec 后必须 RestoreExec |
| **VCC clear** | 循环中 v_add_co 前必须 clear_vcc |

### 11.2 遇到的关键问题与解决方案

#### 问题 #1：GPU 频繁硬挂 (Hard Hang)

**时间**: 3/25 上午
**现象**: `test_gpu_gemm_tn` 触发 GPU page fault at 0x0，每天 6+ 次强制重启
**影响**: 开发完全无法进行

**根因 A — LICM 死代码**:
- LICM 将 hoisted 指令 `extend` 到基本块末尾
- 落在 `s_branch` 终结器之后 → 成为不可达死代码
- GPU 执行到其中 → 非法地址 → page fault

**根因 B — Shr raw_asm 泄漏**:
- `get_vreg(rhs)` 覆写 `val_map` → Shr 走 `raw_asm` 路径
- 虚拟寄存器号绕过 regalloc → 访问未映射的物理寄存器

**修复**:
```rust
// LICM: 插入到 terminator 之前, 而不是 extend
block.ops.insert(block.ops.len() - 1, hoisted_op);

// Shr: 预缓存 rhs_val, 防止 get_vreg 覆写
let rhs_val = val_map[&rhs];  // 先缓存
let lhs_reg = get_vreg(lhs);  // 再获取
```

---

#### 问题 #2：SSA RegAlloc 干涉冲突

**时间**: 3/25 下午
**现象**: GEMM 测试 5/10 通过，其余 crash 或 err=inf
**影响**: VGPR 物理寄存器重叠导致计算结果错误

**根因**:
- `max_vgprs = 128` 硬编码 → GEMM 需要 200+ VGPRs
- 同一 VReg 被多个 MVal 映射，interval 未合并

**修复**:
```rust
max_vgprs = 255;  // 放开硬限制（GFX1100 最大 256）
// 同 VReg 多 MVal 时合并活跃区间
if let Some(existing) = intervals.get_mut(&vreg) {
    existing.end = existing.end.max(new_end);
}
```

**结果**: 干涉冲突 26 → 0，通过率 5/10 → 8/10

---

#### 问题 #3：CSE 5 个根因导致错误结果

**时间**: 3/25 下午
**现象**: CSE 启用后 err=inf 或 GPU hang

| # | 根因 | 修复 |
|---|------|------|
| 1 | `MVal(u32::MAX)` 哨兵被用作 CSE key | 跳过含哨兵 MVal |
| 2 | InlineInt 常量未编码进 CSE key | 编码为 `MVal(0xFE000000\|bits)` |
| 3 | CSE 替换 VMov 用 VReg(0) | 用 `mval_to_vreg` 查找正确 VReg |
| 4 | CSE 不清空 barrier 处的 seen table | `block_seen.clear()` at barrier |
| 5 | `lower_from_ssa` 不做 VReg 重映射 | 完整 `rename_op_uses/defs` |

---

#### 问题 #4：DCE 删除 K-loop 指针步进

**时间**: 3/26 上午
**现象**: GEMM err=12.3 (K-loop 不递增，只用第一个 tile 数据)

**根因**: K-loop 指针步进 `v153 += 32` 的新 MVal 无 use → DCE 删除 → K 不递增

**修复**: backward branch 检测 → loop 内 VReg defs 标记为 root-live
```rust
// 检测后向分支（循环）
if target_block_id <= current_block_id {
    // 标记该 block 中所有 VReg defs 为 root-live
    for op in &block.ops {
        for def in op.vreg_defs() { live_set.insert(def); }
    }
}
```

---

#### 问题 #5：KFD VA 复用竞态

**时间**: 3/26 下午
**现象**: 连续运行 10+ 测试后 GPU hang

**根因**: `FREE_MEMORY → ALLOC_MEMORY` 复用相同 VA 时，GPU L2/TLB 未 invalidate

**修复**: BufferPool LRU 缓存 — buffer "释放" 入池复用，VA 映射永不释放
```rust
// buffer "释放" → 入池缓存（不调 KFD FREE）
pool.release(buf);  // 保持 VA 映射
// 下次分配 → 从池中取出（零 syscall）
let buf = pool.acquire(size)?;
```

---

#### 问题 #6：tile_ir K-loop GMEM→LDS 存储缺失

**时间**: 3/28
**现象**: 所有 `test_tg_*` 返回 err=inf (16384/16384 全错)

**根因** (3 层穿透):
1. `emit_lds_read_and_wmma` 的 bulk-load mode (n_col_tiles ≤ 4) **完全没有 ds_store**
2. `lower_gemm` 末尾 `set_opt_level(4)` 覆盖用户 env var → DCE 始终运行
3. DCE 正确识别 buffer_load 的 VGPR 无人使用 → 删除 → K-loop 只读 stale LDS

**因果链**: 缺失存储 → VGPR 死亡 → DCE 删除 load → stale LDS → err=inf

**修复**: K-loop 中补充 `emit_lds_store_graduated` + `skip_optimize` 禁用 SSA 优化
```rust
// K-loop Phase A
emit_coop_load_buffer(&mut k, &gmem_x_1, ...);
emit_lds_read_and_wmma(..., None);
emit_lds_store_graduated(&mut k, &gmem_x_1, ...buf1...);  // ★ 补充
emit_lds_store_graduated(&mut k, &gmem_wt_1, ...buf1...); // ★ 补充
k.wait_lgkmcnt(0); k.s_barrier();
```

**铁律**: tile_ir 内核必须 `skip_optimize` — SSA passes 对手工优化的 K-loop 有破坏性副作用

---

#### 问题 #7：k48 GPU Hang — Coop Load 地址分解 Bug

**时间**: 3/29
**现象**: 128×128 tile_k=48 即使 0 spill 也 GPU hang

**根因**: cooperative load 的 tid→(row,col) 分解使用 bitwise AND/shift：
```rust
x_row_in_tile = tid >> cpr_shift   // tid / chunks_per_row
x_col_chunk = tid & (cpr - 1)     // tid % chunks_per_row
```
**这只在 `chunks_per_row` 是 2 的幂时正确！**

k48: `chunks_per_row = 48*2/16 = 6` → 非 2^n → tid=2 就已错误 (row=1 应为 0)

**修复**: 添加编译时 assert + 移除 k48 配置
```rust
assert!(chunks_per_row.is_power_of_two(),
    "chunks_per_row={} must be power of 2", chunks_per_row);
```

---

#### 问题 #8：WGP Mode Page Fault

**时间**: 3/29
**现象**: WGP 模式最小 probe 内核（纯 barrier + store）也触发 GPU hang

**dmesg 证据**:
```
sq_intr: error, type 2, priv 1  ← CWSR 特权 wave 错误
page fault: Faulty client = TCP (0x8), PERMISSION_FAULTS = 0x5
MES might be in unrecoverable state → MODE1 reset
VRAM is lost due to GPU reset!
```

**根因**: CWSR (Context Wave Save/Restore) 与 WGP 模式不兼容
- CWSR buffer 按 CU mode 计算 (3072 waves × 12B)
- WGP mode 下 wave 数量和布局不同 → CWSR 地址越界 → page fault

**解决方案**: 放弃 WGP mode, 专注 CU mode 优化
- 候选方案 B: `amdgpu.cwsr_enable=0` (需重启验证)
- CU mode 79 TF 已匹敌 Triton

---

#### 问题 #9：VGPR Gap Reclaim 风险评估

**时间**: 3/29
**现象**: 128×128 k32 使用 254 VGPRs（满载），k64 有 64 spills

**方案**: 实现 gap reclaim — 对齐跳过的间隙 VGPRs 回收到 FreePool

```rust
// 对齐分配时跳过的间隙 VGPRs → 回收
let gap_start = hwm;
let aligned = (hwm + alignment - 1) & !(alignment - 1);
for v in gap_start..aligned {
    free_pool.push(v);  // 回收间隙
}
hwm = aligned + count;
```

**效果**:
- 128×128 k32: 254 → 239 VGPRs（节省 15）
- 128×128 k64: 64 spills → 0 spills
- 所有配置零 hang，正确性不变
