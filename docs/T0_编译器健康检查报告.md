## 1. 编译管线审计

```
T0Kernel (compile.rs)
    ↓ allocate_registers() → RegAlloc
    ↓ AsmEmitter.emit_kernel() → GCN 文本汇编
    ↓ clang -target amdgcn-amd-amdhsa -mcpu=gfx1100 → .o
    ↓ ld.lld → .hsaco (ELF Code Object)
    ↓ KFD: GpuKernel::load() → GPU dispatch
```

| 阶段 | 实现 | 文件 | 状态 |
|------|------|------|------|
| T0Kernel → Op 序列 | 80+ 便捷方法 | compile.rs (945行) | ✅ 完整 |
| Op → GCN 文本 | ~60 Op 变体全覆盖 | asm_emitter.rs (791行) | ✅ 无 catch-all 静默跳过 |
| 寄存器分配 | LinearScan 分配器 | regalloc.rs (275行) | ⚠️ 基础 (无溢出处理) |
| 指令调度 | GFX1100 延迟表 | schedule.rs (376行) | ✅ 基础可用 |
| ELF 生成 | clang + ld.lld | compile.rs L769-945 | ✅ 依赖 ROCm LLVM |

> [!IMPORTANT]
> **寄存器分配器无溢出处理** — 如果内核使用超过 128 VGPR 或 106 SGPR，分配器不会报错也不会 spill，直接产生错误代码。
> GEMM 内核 (gemm_gen) 手动管理寄存器，不依赖 regalloc，因此不受影响。

## 4. 发现的问题

### 4.1 🔴 P0 — 影响正确性

| # | 问题 | 详情 |
|---|------|------|
| 1 | **`lower_simple()` 硬编码 `kernarg_size=40`** | [dsl_lower.rs:L225](file:///home/geis/tprime_optimized/rust-port/tprime-brain/src/OCPA/opensource/src/t0/dsl_lower.rs#L225) — 所有通过 `lower_simple` 的内核都被标记为 40 字节 kernarg，但实际大小各不相同。如果 HSA runtime 使用此值分配 kernarg 缓冲区，可能导致越界读取或错误。 |

## 7. 结论与优先行动

### 编译器整体评估

| 维度 | 评分 | 说明 |
|------|:----:|------|
| **API 完整度** | 9/10 | 80+ 方法覆盖 ISA 关键指令，WMMA、LDS、wave reduce 齐全 |
| **DSL 覆盖度** | 8/10 | 30 个 Op 已映射，仅 SumReduce/Scale 缺失 |
| **代码生成正确性** | 7/10 | clang+ld.lld 管线可靠，但缺乏单元测试 |
| **寄存器分配** | 5/10 | 简单有效但无溢出保护 |
| **测试覆盖** | 1/10 | **零测试** — 最大风险 |

### 优先行动（建议顺序）

1. **修复 `lower_simple` kernarg_size 硬编码** — 改用 `kernel.kernarg_size()`
2. **添加 `Scale` 和 `SumReduce` DSL lower**
3. **写 3 个关键路径测试**:
   - `scale` 内核: 100 个 f32 乘以 2.0，验证结果
   - `residual_add`: a[i] + b[i] 验证
   - `gemm_forward`: 小矩阵 (4×4) CPU 对比
4. **寄存器溢出检测** — 至少在编译时 panic 而非静默错误
