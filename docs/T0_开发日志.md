## 阶段一：初始发布（v0.1.0）

**标志性 commit**: `Initial release: T0 GPU kernel compiler & KFD bare-metal runtime`

### 交付物

| 组件 | 文件 | 描述 |
|------|------|------|
| **ISA 编码器** | `rdna3_asm.rs` (~3500 行) | GFX1100 全指令编码（VOP1/VOP2/VOP3/SMEM/FLAT/WMMA） |
| **ELF 生成器** | `rdna3_code_object.rs` (~1300 行) | AMD HSA ELF Code Object 生成 |
| **KFD 运行时** | `kfd/mod.rs` (~2600 行) | 裸金属 GPU 驱动：AQL 队列、VRAM 管理、零拷贝内存 |
| **T0 编译器** | `t0/` (11 文件) | 基础编译管线：IR → RegAlloc → Schedule → ASM → ELF |
| **内核模板** | `t0/math.rs` (~7800 行) | 70+ 预定义 GPU 内核模板 |
| **GEMM 生成器** | `t0/gemm_gen.rs` | 基础参数化 bf16 WMMA GEMM |

### 架构特点

- 纯 Rust 实现，**零外部依赖**
- 绕过 HIP/ROCm 用户态库，直接通过 `/dev/kfd` 与 GPU 通信
- 调度延迟 ~1-2μs（HIP: 10-50μs）

### 成果

| 矩阵尺寸 | T0 (TFLOPS) | rocBLAS | 比例 |
|----------|-------------|---------|------|
| 1024×1024×1024 | 45.40 | 27.89 | **163%** |
| 1024×1024×4096 | 58.84 | 29.94 | **197%** |
| 4096×4096×4096 | 67.30 | 58.72 | **115%** |

### 优化技术

- 参数化生成器（单一 `generate()` 覆盖所有配置）

- Single-Dispatch Split-K（sk=1~16）
- LDS 双缓冲 + Bank Conflict Padding
- Dual Grid Layout（M-on-X / N-on-X 自适应）
- Auto-Select + K-Clamp

## 阶段三：Block DSL + SSA 编译器重构

> **开发时段**: 约 3 月中旬
> **目标**: 从手写内核模板（math.rs）迁移到编译器自动代码生成

### 新增模块

| 文件 | 行数 | 作用 |
|------|------|------|
| `block_dsl.rs` | ~3000 | Triton 风格 BNode DAG（45+ 节点类型） |
| `block_dsl_to_ssa.rs` | ~1800 | BNode → TileFunc SSA 翻译 |
| `ssa_ir.rs` | ~2500 | MachFunc SSA 表示（MVal/PhiNode/BasicBlock） |
| `opt_passes.rs` | ~3200 | 13-pass SSA 优化管线重写 |
| `domtree.rs` | ~550 | Dominator Tree 实现 |
| `ssa_regalloc.rs` | ~900 | SSA RegAlloc + Register Spill |
| `tile_ssa.rs` | ~2100 | Tile-level SSA IR + TensorLayout 类型系统 |
| `tile_ssa_lower.rs` | ~2300 | TileFunc → T0Kernel lowering |
| `cost_model.rs` | ~700 | GFX1100 精确成本模型（穷举 1040 候选） |
| `hw_probe.rs` | ~1700 | 40+ 指令延迟穷举探测 |
| `latency_model.rs` | ~550 | 实测延迟模型 |
| `isa_probe.rs` | ~1000 | ISA 编码自动验证 |

### 关键架构决策

1. **compile() 委派 compile_via_ssa()**：
   - `block_dsl.rs:911-912` — 所有编译统一走 SSA 管线
   - 保留 `compile_legacy()` 仅供 A/B 对比测试

2. **SSA 优化 4 阶段管线**：
   - Phase A: CF + DCE + CProp + CSE(DomTree) + InstrCombine + LICM
   - Phase B: loop_unroll + strength_reduce
   - Phase C: iterative AlgSimp + DCE
   - Phase D: waitcnt + scheduling

3. **SSA RegAlloc 默认启用**：
   - `compile.rs:65 use_ssa_regalloc: true`
   - 含 spill/reload 支持（LDS scratch 自动管理）

4. **math.rs 冻结**：
   - 概念上被 BlockKernel DSL 替代
   - 因 Ignis 多处引用暂不可移除

## 混乱根源分析

### 问题总结

| 问题 | 具体表现 | 影响 |
|------|---------|------|
| **三层重复结构** | `src/t0/`(旧) + `src/OCPA/`(副本) + `opensource/`(新) | AI 审查时将旧代码当主路径 |
| **rdna3_asm.rs 5 份副本** | 大小不完全相同（142,666 vs 142,648） | 不清楚哪份是权威版本 |
| **math.rs 被替代但不能删** | Ignis 仍引用，BlockKernel DSL 已替代 | 误以为 math.rs 是主开发路径 |
| **缺少开发日志** | 只有对比文档，没有过程记录 | 无法追溯决策原因 |
| **文档更新滞后** | `opensource/docs/` 里的文档停在第五轮（90%） | 进展未同步 |
| **旧代码仍在编译** | Layer A/B 的 `t0/` 和 `rdna3_asm.rs` 仍被宿主项目编译 | 编译噪音，维护负担 |

### 根本原因

1. 项目从「手写内核 → 编译器自动生成」的范式转变中，**旧代码没有被清理**
2. 开源版 `opensource/` 作为独立 crate 发展，但**没有替换掉宿主项目中的旧副本**

## 当前状态快照（2026-03-25）

### 权威代码源

**`src/OCPA/opensource/`** — 独立 crate `t0-gpu`

### 编译管线

```
BlockKernel DSL (block_dsl.rs, 45+ BNode)
  → compile() → compile_via_ssa() (block_dsl_to_ssa.rs)
    → TileFunc SSA (tile_ssa.rs)
      → lowering (tile_ssa_lower.rs / tile_ir.rs)
        → T0Kernel (ir.rs, 90+ Op)
          → opt_passes (13-pass SSA)
            → ssa_regalloc (含 spill)
              → asm_emitter → llvm-mc → HSA ELF
                → KFD Runtime → 🚀 GPU
```
